Send a SCOUT First: Pre-hoc Reasoning for Adaptive Detector Allocation in Prompt-Injection Defense

Prompt-injection defense should not commit every request to one fixed detector. SCOUT routes each input through a detector pool using predicted detector reliability and latency.

Abstract / Overview

Detector allocation instead of detector commitment

Prompt-injection detectors are heterogeneous: each is fast or accurate on a different slice of requests, and none is best everywhere.

SCOUT treats defense as a routing problem. For each input, it predicts how each detector is likely to behave, runs a predicted-reliable light pool, and escalates to an LLM judge only when the vote remains uncertain.

The system exposes one threshold, tau, to control the safety-latency trade-off. Raising the threshold increases judge usage; lowering it trusts the light pool more often.

  • Per-input detector allocation instead of a fixed single-detector pipeline.
  • Fingerprint-based prediction of detector correctness and latency from similar past inputs.
  • One threshold for operating-point control across safety, benign utility, and wall-clock.
  • SCOUT-450 plus transfer results on BIPIA, IPI, and IHEval.

Method

Fingerprint, predict, route

SCOUT builds a behavioral fingerprint for each detector, predicts per-input reliability and latency, then decides whether the light pool is confident enough to answer.

A retrieved Anchor-400 neighborhood provides detector fingerprints for inputs that look similar to the current request. A small predictor reads those fingerprints and emits a correctness and latency estimate for each detector.

The router keeps only detectors predicted to be reliable, aggregates them with trust weights, and compares the vote confidence against the threshold. If the vote is not decisive, SCOUT escalates to the LLM judge.

Results

SCOUT-450 and cross-benchmark transfer

On SCOUT-450, the headline operating point at tau = 0.875 reaches the lowest total ASR and the highest accuracy among the reported systems while using about 0.6x the total wall-clock of an always-on GPT-4o judge.

Data

SCOUT-30K, Anchor-400, and SCOUT-450

SCOUT uses separate data roles for detector training, predictor supervision, fingerprint retrieval, and evaluation.

  • SCOUT-30K: 29,551 hindsight-distilled sample-detector examples for predictor supervision.
  • Anchor-400: 400 examples used for kNN retrieval and detector trust priors.
  • SCOUT-450: 450 evaluation samples, with 255 attacks and 195 benign cases, enriched for harder agent-facing injections.

The page focuses on the evaluation benchmark, but the routing system depends on keeping these roles disjoint.
